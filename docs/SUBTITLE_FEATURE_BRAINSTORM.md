# 字幕能力头脑风暴记录

更新时间：2026-06-30

## 背景

用户决定暂缓 B 站 / YouTube 在线源需求，当前新的方向是：

- 支持本地视频挂载外部字幕
- 支持字幕快 / 慢 1 秒来对齐视频
- 更高阶：支持实时字幕与实时翻译

产品边界仍然保持轻量：核心围绕视频播放、观看记录、观看时间记录、多源读取与导入播放。

## 当前项目基础

当前播放器使用 ArtPlayer。

有利条件：

- `Player.tsx` 已经开启 `subtitleOffset: true`
- ArtPlayer 官方支持 `vtt`、`srt`、`ass` 字幕
- ArtPlayer 官方提供 `subtitle` 配置，支持 `url`、`type`、`encoding`、`style`、`onVttLoad`
- 当前主进程已有本地 HTTP Range 视频流服务，可以扩展一个字幕文本转发接口
- 当前数据结构已有按视频唯一 key 保存进度和时间轴笔记的模式，字幕配置也可以按视频 key 保存

需要补齐：

- 字幕文件选择对话框
- 字幕文件本地 HTTP 访问 URL
- 每个视频关联字幕路径、字幕偏移、字幕开关
- 自动匹配同名字幕
- 字幕路径迁移或失效提示

## 外部字幕 MVP

建议第一版只做轻量能力：

- 手动选择字幕文件
- 支持 `.srt`、`.vtt`，可选 `.ass`
- 自动保存该视频挂载的字幕路径
- 字幕偏移支持 `-5s ~ +5s`
- 提供快捷按钮：
  - 字幕提前 1 秒
  - 字幕延后 1 秒
  - 重置偏移
  - 关闭字幕
- 后续打开同一视频自动恢复字幕与偏移

建议优先级：

1. 手动挂载字幕
2. 保存字幕路径与 offset
3. 快 / 慢 1 秒按钮和快捷键
4. 同名字幕自动匹配
5. 多字幕轨道选择

## 自动匹配策略

本地视频：

- 如果视频是 `A.mp4`
- 自动查找同目录下：
  - `A.srt`
  - `A.vtt`
  - `A.ass`
  - `A.zh.srt`
  - `A.en.srt`

匹配到一个时自动挂载。

匹配到多个时：

- 不自动弹窗打扰
- 在播放器字幕菜单里显示可选项
- 默认优先中文，其次英文，其次第一个

WebDAV / AList：

- 第一版可以先不做自动匹配
- 后续如果目录扫描能拿到字幕文件，再按同名规则补上

## 数据结构草案

```ts
interface SubtitleAttachment {
  path: string;
  name: string;
  type: 'srt' | 'vtt' | 'ass';
  offset: number;
  encoding?: string;
  enabled: boolean;
  autoMatched?: boolean;
  lastUsedTime: number;
}

interface AppDataStore {
  subtitles?: Record<string, SubtitleAttachment>;
}
```

其中 `Record` 的 key 复用当前视频 key，也就是本地绝对路径 / WebDAV URL / 未来在线源唯一 ID。

## 实现路线

### 第一阶段：外部字幕

主进程：

- 增加 `dialog:selectSubtitleFile`
- 增加 `subtitle:getUrl`
- 在现有本地 HTTP 服务增加 `/subtitle?path=...`
- 正确返回 `text/vtt` / `application/x-subrip` / `text/plain`
- 对路径做 base64 编码传参，复用现有视频流服务的安全处理模式

渲染进程：

- 在 `storage.ts` 增加 `subtitles` 存储字段
- 在播放器 UI 增加字幕控制入口
- `Player.tsx` 初始化 ArtPlayer 时传入 `subtitle`
- 切换视频时读取该视频字幕配置
- 监听字幕偏移变化并保存

### 第二阶段：自动匹配

- 本地源扫描时识别字幕文件，但不展示为视频节点
- 或播放时由主进程按视频路径查找同名字幕
- 自动挂载后写入 `subtitles`
- 如果字幕路径不存在，播放器显示轻量提示并禁用该字幕记录

### 第三阶段：多字幕

- 支持同一视频多个字幕轨道
- 右侧栏或播放器设置中选择字幕
- 未来可以支持原文 + 翻译双字幕

## 实时字幕可行性

实时字幕分两条路线：

### 云端实时转写

优点：

- 准确率高
- 实时性更好
- 不增加本地安装体积
- 可支持多语言

缺点：

- 需要 API key
- 有费用
- 需要联网
- 视频音频需要发送到云端，必须明确隐私提示
- 需要从播放器音频流中采样 PCM，再送到实时转写服务

适合定位：

- 高级可选功能
- 默认关闭
- 由用户自己配置 API key

### 本地实时转写

可选方案：

- whisper.cpp
- faster-whisper
- WhisperLive

优点：

- 隐私更好
- 离线可用
- 不产生云端调用费用

缺点：

- 模型文件体积大
- CPU / GPU 占用高
- 打包复杂度明显增加
- 实时效果依赖机器性能
- 会让软件变重

适合定位：

- 不建议作为主线内置
- 可做成后续“可选扩展包”

## 实时翻译可行性

实时翻译比实时字幕更重，因为链路变成：

```text
视频音频 -> 实时 ASR -> 文本分段 -> 翻译 -> 字幕展示
```

或者：

```text
视频音频 -> 实时翻译服务 -> 译文字幕 / 译音频
```

风险点：

- 延迟不可避免，通常需要 1~3 秒缓冲
- 分句不稳定，实时字幕可能先错后改
- 翻译上下文不足时质量会波动
- 长视频费用需要考虑
- 需要设计缓存，避免同一视频重复花钱

建议路线：

1. 先做“已有字幕翻译”
   - 用户挂载英文字幕
   - 一键生成中文字幕
   - 保存为本地派生字幕
   - 这是更轻、更稳定、更符合学习场景的方案
2. 再做“离线生成字幕”
   - 对整段视频生成一次字幕文件
   - 生成后像普通外部字幕一样使用
3. 最后再考虑“播放中实时字幕 / 实时翻译”
   - 做成高级实验功能
   - 明确 API key、费用、隐私、延迟

## 推荐结论

最适合当前产品的路线：

1. 立即可做：外部字幕挂载 + 偏移调整
2. 很值得做：同名字幕自动匹配
3. 谨慎做：已有字幕一键翻译生成新字幕
4. 暂缓：真正实时字幕
5. 更暂缓：真正实时翻译

原因：

- 外部字幕与偏移调整是播放体验增强，轻量、稳定、改动面小
- 自动匹配字幕能大幅减少手动操作
- 已有字幕翻译能服务学习场景，但不会拖累播放主流程
- 实时字幕 / 实时翻译技术上可行，但成本、隐私、复杂度都明显高一档
